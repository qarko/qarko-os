# QARKO OS: Codex 수준 Hermes CLI 래퍼 설계

작성일: 2026-05-25

## 목표

QARKO OS의 핵심 목표는 Hermes CLI를 터미널 없이 사용할 수 있게 만드는 Windows 데스크톱 워크벤치다. 사용자는 Codex Windows 앱을 쓰듯이 프로젝트를 만들고, 프로젝트 안에서 Hermes와 대화하고, Hermes가 실행하는 작업의 상태와 로그를 확인할 수 있어야 한다.

QARKO OS는 사업 아이템이나 1인 MVP에 사용될 수 있지만, 제품 자체를 특정 사업 범위에 묶지 않는다. 기본 목적은 다음 한 문장으로 고정한다.

> QARKO OS는 Hermes CLI를 Codex Windows 앱 수준의 데스크톱 경험으로 감싸는 앱이다.

## 이번 범위

이번 설계와 구현의 우선 범위는 로컬 Windows 환경에서 Hermes CLI를 안정적으로 감싸는 것이다.

반드시 포함한다:

- 프로젝트별 Hermes 세션 유지
- 터미널/CLI 실행 감싸기
- 작업 로그 저장과 표시
- 긴 작업 중 상태 표시
- Codex Windows 앱과 유사한 작업실 UI
- Hermes 설치, 인증, 모델 설정을 앱 안에서 처리

이번 범위에서 제외한다:

- SSH 원격 실행
- QARKO 자체 클라우드 실행 환경
- 커뮤니티/플러그인 마켓 고도화
- 결제/계정/팀 기능

SSH는 나중에 사용자가 본인의 서버 정보를 등록해서 원격 Hermes Runner로 사용할 수 있도록 확장한다. 지금은 내부 구조가 그 확장을 막지 않도록만 고려한다.

## 핵심 제품 구조

QARKO OS의 기본 단위는 프로젝트다. 프로젝트는 단순 문서 묶음이 아니라 Hermes 실행 상태를 가진 작업실이다.

```text
Project
  - id
  - name
  - workspace folder
  - active Hermes session id
  - selected model/provider
  - permission mode
  - execution status
  - chat messages
  - execution logs
  - artifacts
  - changed files
```

프로젝트 A와 프로젝트 B는 서로 다른 Hermes 세션과 로그를 가져야 한다. 사용자가 프로젝트 A에서 대화를 이어가면 항상 프로젝트 A의 Hermes 세션으로 이어져야 하고, 프로젝트 B로 전환하면 프로젝트 B의 Hermes 세션을 사용해야 한다.

## 프로젝트별 세션 유지

Hermes CLI가 세션 id를 제공하면 QARKO OS는 해당 값을 프로젝트 실행 상태에 저장한다. 이후 같은 프로젝트에서 사용자가 새 메시지를 보내면 새 대화를 시작하지 않고 저장된 Hermes session id로 resume한다.

원칙:

- 프로젝트 하나는 기본적으로 Hermes 세션 하나를 유지한다.
- 사용자가 명시적으로 새 세션을 만들 때만 세션을 교체한다.
- 세션 id는 로컬 실행 상태에만 저장한다.
- 세션 id는 서버 동기화나 피드백 전송에 포함하지 않는다.
- Hermes가 세션 id 오류를 반환하면 사용자에게 복구 선택지를 제공한다.

복구 선택지:

- 같은 프로젝트에서 새 Hermes 세션 시작
- Hermes 상태 다시 확인
- 마지막 사용자 메시지 다시 실행

## 터미널/CLI 실행 감싸기

QARKO OS는 사용자가 터미널을 직접 보지 않아도 Hermes CLI를 사용할 수 있게 해야 한다. 내부적으로는 Hermes CLI를 실행하지만, UI에서는 대화, 상태, 로그, 오류로 해석해서 보여준다.

실행 흐름:

```text
사용자 입력
  -> 프로젝트 컨텍스트 구성
  -> Hermes 실행 요청 생성
  -> Hermes CLI 실행
  -> stdout/stderr/status 수집
  -> 채팅 메시지, 작업 로그, 상태로 분리 저장
  -> UI 업데이트
```

QARKO OS가 숨겨야 하는 것:

- 복잡한 PowerShell 명령
- 환경 변수 설정 명령
- Hermes 설치 스크립트 내부 과정
- 불필요한 raw stderr

QARKO OS가 보여줘야 하는 것:

- 현재 실행 중인 단계
- Hermes가 요청을 처리 중인지 여부
- 인증/모델/설치 문제
- 사용자가 직접 해결해야 하는 조치
- 명령 실패 이유와 다음 행동

## 작업 로그

작업 로그는 채팅 메시지와 분리한다. 채팅은 사용자와 Hermes의 대화이고, 작업 로그는 QARKO OS가 Hermes 실행 과정을 기록하는 운영 로그다.

로그 이벤트 예시:

- Hermes 실행 시작
- 세션 resume 사용
- 모델/provider 확인
- 인증 필요
- 명령 실행 중
- stdout 수신
- stderr 수신
- Hermes 완료
- Hermes 오류
- 사용자가 취소

로그는 프로젝트별로 저장하며, 최근 로그는 오른쪽 패널에 표시한다. 상세 로그는 별도 화면에서 볼 수 있게 한다.

## 긴 작업 중 상태 표시

Hermes 작업이 오래 걸릴 수 있으므로 QARKO OS는 사용자가 멈춘 것으로 오해하지 않게 해야 한다.

필수 상태:

- 준비됨
- 실행 대기
- 실행 중
- Hermes 응답 수신 중
- 사용자 승인 대기
- 완료
- 오류
- 취소됨

UI 원칙:

- 중앙 대화 영역에는 현재 Hermes가 작업 중임을 명확히 표시한다.
- 오른쪽 패널에는 세부 단계와 최근 로그를 표시한다.
- 오래 걸리는 작업에는 경과 시간을 표시한다.
- 사용자가 취소할 수 있어야 한다.
- 오류 발생 시 기술 로그만 보여주지 말고 다음 행동을 제시한다.

## Codex식 작업실 UI

QARKO OS는 3분할 대시보드가 아니라 Codex Windows 앱처럼 프로젝트 작업실 중심이어야 한다.

기본 레이아웃:

```text
왼쪽: 프로젝트/세션 목록
가운데: 프로젝트 작업실
오른쪽: 접힌 상태의 보조 패널
```

가운데 작업실:

- 프로젝트 제목
- 현재 모델/provider
- 권한 모드
- 사용자 메시지
- Hermes 응답
- 실행 중 상태
- 입력창

오른쪽 보조 패널:

- 실행 로그
- 변경 파일
- 산출물
- 승인 대기
- Hermes 상태
- 피드백

오른쪽 패널은 항상 넓게 차지하지 않는다. 기본은 작게 접힌 레일이며, 사용자가 로그나 변경 파일을 눌렀을 때 필요한 화면이 열린다.

## Hermes 설정과 인증

첫 실행 또는 Hermes 미준비 상태에서는 QARKO OS가 설정 마법사를 제공한다.

설정 마법사 단계:

1. Hermes 설치 확인
2. 미설치 시 고정 버전 설치
3. provider 선택
4. 인증 방식 선택
5. OAuth 또는 API key 연결
6. 모델 선택
7. Hermes doctor/status 확인
8. 테스트 메시지 실행

인증 과정에서 브라우저가 필요한 경우 QARKO OS가 브라우저를 열고, 사용자는 QARKO OS로 돌아와 인증 상태 확인 버튼을 누른다. 터미널 창을 직접 다루게 하지 않는다.

## 권한과 샌드박스

QARKO OS에서는 "샌드박스"라는 어려운 표현 대신 "샌드박스(안전 승인 모드)"라고 표시한다.

기본 권한 모드:

- 보기/초안: 파일 변경 없이 계획과 답변만 생성
- 샌드박스(안전 승인 모드): 프로젝트 폴더 안에서만 작업하고 위험 작업은 승인 필요
- 자동 실행: 사용자가 허용한 범위에서 반복 작업 자동 실행

베타 기본값은 샌드박스(안전 승인 모드)로 둔다.

## SSH 확장 방침

SSH는 가능하지만 이번 구현 범위에서는 보류한다. 나중에 사용자가 본인의 서버 SSH 정보를 등록해서 원격 환경에서 Hermes를 실행할 수 있게 한다.

나중에 추가할 구조:

```text
Runner
  - Local Hermes Runner
  - SSH Hermes Runner
```

이번 단계에서는 Local Hermes Runner를 먼저 안정화한다. 단, 실행 엔진의 인터페이스는 나중에 SSH Runner를 추가할 수 있도록 분리한다.

## 성공 기준

베타 테스트 전 성공 기준:

- 사용자가 터미널을 직접 열지 않고 Hermes를 설치/설정할 수 있다.
- 사용자가 프로젝트를 만들고 그 안에서 Hermes와 계속 대화할 수 있다.
- 같은 프로젝트에서 대화가 새 세션으로 끊기지 않는다.
- Hermes 실행 중 상태가 명확히 보인다.
- 작업 로그가 프로젝트별로 남는다.
- 오류가 발생하면 사용자가 다음에 무엇을 해야 하는지 알 수 있다.
- UI가 Codex Windows 앱처럼 작업실 중심으로 느껴진다.

## 다음 구현 계획

1. Runner 아키텍처를 코드에 반영한다.
2. Hermes 실행 상태 모델을 프로젝트별로 정리한다.
3. 채팅 메시지와 실행 로그를 분리한다.
4. 긴 작업 상태와 취소 흐름을 구현한다.
5. 오른쪽 패널을 Codex식 접힘/확장 패널로 정리한다.
6. Hermes 설정 마법사를 실제 사용 흐름 기준으로 다시 검수한다.
7. 베타 설치 파일을 다시 빌드하고 실제 Hermes 연결 테스트를 진행한다.
